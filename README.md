# UX/CX Research — Карта компетенций

Интерактивный инструмент оценки компетенций UX/CX исследователей.
Поддерживает два профиля (UX / CX) и три роли оценки (самооценка, лид команды, лид продукта).

**Продакшен:** https://uxcx-assessment.vercel.app

## Запуск

```bash
npm install
npm run dev
```

Приложение будет доступно по адресу [http://localhost:3000](http://localhost:3000).

## Профили исследователей

На главной странице выбирается профиль, который влияет на **агрегацию** итогового уровня (per-competency скоринг одинаков для обоих):

### UX Research (по умолчанию)

Quant-компетенции (`hard_quant_design`, `hard_quant_analysis`) влияют на итог по правилам гейтинга:

| quantLevel (min из двух quant) | Поведение |
|------|-----------|
| 1 | Quant участвует в Craft; **overall жёстко ограничен ≤ 2** |
| 2 | Quant **исключается** из агрегации осей (не влияет ни вверх, ни вниз) |
| ≥ 3 | Quant участвует в Craft наравне с остальными |

### CX Research

Компетенция `hard_execution` **исключается** из агрегации осей и итогового уровня (но остаётся в результатах по компетенциям).

## Роли оценки

| Роль | Описание |
|------|----------|
| **Самооценка** | Исследователь оценивает себя |
| **Лид команды** | Лид исследователей оценивает сотрудника |
| **Лид продукта** | Продуктовый лид оценивает сотрудника |

Каждая роль заполняет одинаковый набор из 114 утверждений. Результаты всех ролей отображаются на одном экране с наложением на радар-чарте, сравнительной таблицей и выявлением «слепых зон».

## Как работает скоринг

### Уровень компетенции

Для каждого `level_target` в рамках компетенции проверяется условие прохождения (считаются только отвеченные вопросы, пропуски игнорируются):

- **Leadership** (строгое правило): все отвеченные ≥ 4
- **Craft / Impact**: ≥ 70% отвеченных ≥ 4

Компетенция получает **максимальный level_target**, прошедший проверку. Если ни один — уровень 1.

### Уровень оси

Среднее арифметическое `achievedLevel` компетенций внутри оси (с учётом profile-фильтрации), округлённое вниз. Диапазон 1–5.

### Потолок данных (axis ceiling)

Если у оси максимальный `level_target` в CSV ниже 5 (например, Craft — только уровни 1–3), и ось достигла своего потолка — она **не ограничивает** общий уровень. Это позволяет Leadership (у которой есть уровни 4–5) «поднять» итоговый результат.

### Общий уровень

Наибольшее N (1–5), при котором:
- минимум 2 из 3 effective axis levels ≥ N
- ни один effective axis level < N − 1

После этого применяется UX hard cap: если `quantLevel === 1`, то `overallLevel ≤ 2`.

### Слепые зоны (мультиролевой режим)

Когда заполнены оценки нескольких ролей, система выявляет компетенции, где самооценка расходится с внешней оценкой на ≥ 2 уровня.

## Страница результатов

- **Общий уровень** с названием
- **Радар-чарт** по осям (с наложением ролей)
- **Карточки осей** со средним уровнем и прогрессом
- **Слепые зоны** (при наличии нескольких ролей)
- **Интерпретация профиля** (текстовые блоки на основе комбинации осей)
- **Таблица компетенций** с достигнутым уровнем
- **План развития** с рекомендациями по слабым компетенциям
- **Утверждения ниже 4** — список всех вопросов, где оценка 1–3

## Совместная оценка (Supabase)

Три разных человека могут оценить исследователя со своих устройств через общую сессию.

### Как настроить

1. Создайте бесплатный проект на [supabase.com](https://supabase.com)
2. В **SQL Editor** выполните содержимое файла `supabase-schema.sql`
3. Скопируйте `.env.local.example` в `.env.local`:
   ```bash
   cp .env.local.example .env.local
   ```
4. Заполните переменные в `.env.local` значениями из **Settings → API**:
   - `NEXT_PUBLIC_SUPABASE_URL` — Project URL
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY` — anon / public key
5. Перезапустите `npm run dev`

### Как пользоваться

1. **Исследователь** нажимает «Создать сессию» → получает 6-символьный код
2. Делится кодом с **лидом команды** и **лидом продукта**
3. Каждый участник: открывает приложение → «У меня есть код» → вводит код → выбирает роль → проходит оценку
4. Ответы автоматически сохраняются в Supabase
5. Любой участник может посмотреть сравнительные результаты

> Без Supabase приложение работает в локальном режиме (всё хранится в localStorage).

## Как редактировать банк вопросов

Файл: `public/data/assessment_items.csv`

```
item_id,competency_id,level_target,statement
PS_01,hard_problem_setting,1,"Я умею формулировать цель и задачи исследования"
```

- **item_id** — уникальный идентификатор (должен быть уникален в файле)
- **competency_id** — ID компетенции из `competency_model.json`
- **level_target** — целевой уровень (1–5)
- **statement** — текст утверждения в двойных кавычках

При добавлении новой компетенции: добавьте её в `competency_model.json` и `axis_mapping.json`, затем создайте вопросы в CSV.
